# 要件定義書：検査データ分布比較システム（MVP Ver 1.0）

## 1. 目的・前提

### システム目的
製造工程の品質管理プロセスにおける、異なる測定データセット間の統計的分布を視覚的かつ定量的に効率よく比較可能にすること。これにより、エンジニアがデータのばらつきや偏りを迅速に特定し、根本原因分析（RCA）と工程最適化を行い、結果として製造コスト削減を実現する。

### 用語集
*   **シグマプロット**: 本ドキュメントではQ-Qプロット（Quantile–Quantile Plot）のことであり、2つのデータセットの分布形状を同一スケールで比較するためのグラフである。

## 2. 業務要件

### 対象業務一覧
工程間品質管理データの取得、分析、および傾向把握。

### 利用者・担当者
品質管理部門のエンジニア。

### 業務課題とKPI、対応方針
*   **業務課題**: 工程データがどの分布パターンを持つべきか（理想的な状態）から逸脱している場合、その原因特定に時間を要し、プロセス改善サイクルが遅延する。
*   **KPI (重要業績評価指標)**: データアップロードから分析完了までの時間短縮、および異常傾向の早期発見率向上。
*   **対応方針**: 必要なデータ取り込みと統計解析表示をGUI上でワンストップで完結させ、エンジニアが直感的に「問題点」を把握できるようにする。

## 3. 機能要件

### 機能一覧（業務課題から導出された最小機能）
1.  **データ管理・アップロード機能**: CSVファイルによる測定値データの取り込み。
2.  **比較設定・実行機能**: ユーザーが分析したい2つのカラムをGUI上で選択し、解析を実行する。
3.  **分布可視化表示機能（メイン）**: Q-Qプロットとヒストグラムを同時に描画し、データ間の統計的な差異を一目で比較可能にすること。
4.  **共通ログ記録機能**: システムの利用履歴やエラー発生時を自動でシステムログとして記録する。

### 業務課題への紐づけ（最小単位）
*   **【分布の視覚化】**: Q-Qプロットとヒストグラムを表示することで、データの偏りやばらつきが「理想的な状態」から逸脱していることを可視化し、異常な傾向を特定できる。（業務課題解決に必須）

### 3.1 データ処理仕様 (入力)
*   **データ源**: ローカルのCSVファイル。
*   **内容**: 複数の数値測定値を含む単一ファイル。カラムには「日付」「ロットID」などのメタ情報が含まれることが想定されるが、分析に使用するのは選択された純粋な数値カラムのみとする。
*   **規模**: 最大数千レコードを処理可能であること。

### 3.2 出力仕様・外部連携
*   本システムは解析結果や元データを永続保存しないため、出力データ形式はGUI描画（画像）に限定する。また、外部連携機能は不要とする。

## 4. データ要件とエンティティ定義 (重要)
このシステムでは「一時的な測定値データ」を主要な業務エンティティとして扱います。永続的なデータベース接続や保存は行いませんが、データライフサイクル管理のため、仮想的なエンティティとして定義します。

### 業務エンティティ一覧
| エンティティ名 | 説明 | データ保持期間 | CRUD / 機能 | 状態（Status） | 管理機能 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **測定値データ** (MeasurementDataset) | アップロードされたCSVから取り込まれ、解析対象となる一時的な数値の集合。 | セッション終了時まで | 一覧表示 / 詳細参照 / 検索（アップロード元ファイル名による） | `未処理` → `処理中` → `完了` | N/A (ログ記録に留める) |

### エンティティの状態遷移
*   **状態**: データ取り込みのライフサイクルに伴うステータス管理が必要です。
    1.  **初期状態**: `未処理`（アップロード直後）
    2.  **進行中**: `処理中`（カラム選択・解析計算が実行されている間）
    3.  **完了**: `完了`（すべてのグラフ描画が完了し、ユーザーに結果が表示された状態）

## 5. 非機能要件
*   **性能（応答時間）**: アップロード〜描画完了までを10秒以内とする。（最重要）。プロセスが完了するまで待機せず、結果を即座に描画すること。
*   **利用人数・セキュリティ**: 同時接続ユーザー数は一人（単独利用者）と想定し、認証機能や認可機能は不要とする。
*   **共通（ログ）**: 以下のイベントが発生した際、システムが自動でタイムスタンプ付きのログを記録する機構が必要です。
    *   ① アップロード実行時 (成功/失敗)
    *   ② カラム選択・解析実行時 (どのカラムセットが選択されたか)
    *   ③ エラー発生時（致命的な処理エラー）

## 6. テスト用利用シナリオ（網羅性検証）
| Test Purpose | 前提条件 | Test Procedure | Expected Result |
| :--- | :--- | :--- | :--- |
| **基本フロー実行確認** | 複数カラムを持つ適切なCSVデータが手元にある。 | 1. ファイルアップロード機能でデータをアップロードする。<br>2. カラム選択機能で2つのカラムをセットで選び、「実行」ボタンを押す。 | Q-Qプロットとヒストグラムが同時に正常に表示されること。処理完了時、ログに「`完了`」ステータスが記録され、成功メッセージが表示されること。 |
| **エラーハンドリング確認** | CSVファイルの一部が数値カラムではない（例：テキスト）。 | 1. 不正なデータを含むCSVをアップロードする。<br>2. 解析実行を試みる。 | システムが処理を開始せず、「指定されたカラムはすべて数値である必要があります」といった具体的なエラーメッセージを表示すること。ログに「`エラー発生`」として記録されること。 |
| **比較対象の変更確認** | 適切なCSVデータがアップロード済み。 | 1. 解析実行後、再度別の2つのカラムを選択し、「再実行」する。 | 新しい選択に基づき、Q-Qプロットとヒストグラムが瞬時に更新表示され、古い結果は新しい結果に置き換わること。ログには「`処理中`」の状態遷移が記録されること。 |
