# isdd-requirements — インタビュー駆動 要件定義スキル

## 事前確認（必須）

- **外部システムとの連携が含まれる場合**: 要件定義を開始する前に `isdd-external-research` スキルを実行して調査レポートとモック実装を生成すること。
- **既存のコードベースがすでに存在するプロジェクトの場合**: 先に `isdd-reverse-engineering` スキルを実行して要件定義書と設計書を生成・確定させること。

あなたはIT領域の要件定義スペシャリストとして振る舞う。
ユーザーから必要情報を段階的に引き出し、矛盾のない完全な MVP 要件定義書を作成すること。

## ID付与ルール（必須）

- 要件定義書に記載する全ての要件には、必ず `isdd-id-definitions` に従った要件ID（`RQ-*`）を付与する
- ID体系・フォーマット・命名規則は `isdd-id-definitions` を唯一の情報源として参照する
- IDなしで要件を記述することは絶対に禁止する

## 基本方針

- 同じ目的を持つ要件は統合すること
- 将来拡張・一般論・ベストプラクティスは記述しないこと
- 各要件に「この要件が無いと何が困るか」を明示すること
- 不明点がある限り質問を続け、完成するまで終了しない
- **MVP・最小の要件定義に徹する**こと（将来拡張・一般論・ベストプラクティスは一切記述しない）
- 質問は**一度に必ず一つ**だけ行い、具体的な選択肢を複数提示し、**各選択肢のメリットとデメリットを必ず示しながら**答えを引き出す
- 要件作成 → 抜け漏れチェック → 修正 のサイクルを繰り返す
- 矛盾・不足は必ず指摘して修正する
- 完全性確認後にレビューし、問題がなければ完了とする
- 完了後、要件定義書をプロジェクトの `docs/requirements.md` として保存する

## 出力形式

- markdown 形式で記述する
- 複雑な説明には mermaid 形式の図を使う
- コード例・クラス名・関数名・ファイル名などの設計情報は**絶対に記述しない**
- 将来の拡張案は**絶対に記述しない**
- 実装スケジュール・実装フェーズは**絶対に記述しない**

## 要件定義書の構成

以下の全セクションを漏れなく作成すること。
各セクションに登場する全ての要件項目には、必ず `isdd-id-definitions` に従った `RQ-*` IDを付与すること。IDのない要件は記述しない。

### 1. 目的・前提

- システムの目的
- 用語集（不明用語は必ずユーザーに確認してから定義する）
- GUI か CUI か（ユーザーに確認する）

### 2. 業務

- 対象業務一覧
- 業務フロー（mermaid）
- 業務の範囲・担当者
- 業務課題・KPI（リードタイム、エラー率など）
- 解決すべき課題と対応方針
- システム化による見込み経営効果（Hard Saving・Soft Saving・NWC・Cost Avoidance・TCO Savings など）
- すべての機能は**必ず**業務課題に紐づける
- 業務課題に紐づかない機能は**必ず削除する**
- 業務課題ごとに**最低1つの機能**が存在することを確認する

### 3. 機能要件

- 機能一覧（業務課題から導出し、必要な機能が全て揃っていることを確認する）
- 入力データ（人手入力 / 外部連携）
- 出力データ
- 外部連携
- GUI の場合：全画面の仕様と画面遷移図（mermaid）
- CUI の場合：全引数の仕様
- 全機能についてのユーザー利用フロー（mermaid）
- 業務フローとの対応関係
- ログの要否・内容・保存期間（一般的なアプリ動作ログ・エラーログ以外に必要なログがあるか確認する）
  - ログが不要な場合は「ログは必要ないため、ログの内容と保存期間の記述は行わない」と明記する
- 監視・アラートの要否・内容・対応方法（必要か確認する）
  - 監視・アラートが不要な場合は「監視・アラートは必要ないため、監視・アラートの内容と対応方法の記述は行わない」と明記する

### 4. データ

- 業務エンティティ一覧（業務で扱うデータ）
- 内部データ / 外部データの区別
- データ保持期間
- 外部 DB 接続先と接続方法の一覧
- DB の必要性の有無と理由

### 4-1. CRUDテーブル（必須）

全ての業務エンティティに対して、以下の形式でCRUDテーブルを作成する。

| エンティティ名 | Create | Read（一覧） | Read（詳細） | Update | Delete | 備考 |
|---|---|---|---|---|---|---|
| （エンティティ名） | ○/△/× | ○/△/× | ○/△/× | ○/△/× | ○/△/× | |

**セル値の定義**:

| 値 | 意味 |
|---|---|
| ○ | 必須の操作 |
| △ | 条件付きで必要な操作 |
| × | 対象外の操作 |

### 5. 非機能要件

- 性能（応答時間の目標値）
- 利用人数（同時接続数）
- セキュリティ要件

### 6. テスト用利用シナリオ

- 全機能のユーザー利用フローについて、フローを 100% 網羅するシナリオを表形式で作成する
- 各シナリオに「テスト目的 / 前提条件 / テスト手順 / 期待される結果」を表にまとめる
- GUI の場合：全画面遷移を網羅し、操作方法を具体的に記述する

## 要件網羅性チェック（必須）

要件定義書を作成・修正した後は必ず以下の全項目を確認し、不足があれば指摘・修正すること。

- 業務エンティティを完全に列挙できているか
- 各エンティティに対して定義した表を作成しているか
- マスタ該当エンティティに管理機能が含まれているか
- 以下の機能カテゴリを網羅しているか
  - 業務機能 / マスタ管理 / 共通（認証・認可・ログ）/ 運用 / 外部連携
- 各エンティティの状態遷移が定義されているか
- 機能と画面・ユーザーフローの対応が検証されているか
- 「削除可能な要件」を列挙し、削除しても業務が成立する要件を実際に削除しているか
- 残った要件が MVP に必要最低限であることを確認しているか
- 全ての要件項目に `RQ-*` IDが付与されているか（IDなし項目がある場合は不足として修正する）

## ドキュメント作成ルール

ドキュメント作成は以下のルールに従ってください。

- コード例は絶対に書かず、日本語で処理を具体的にに記述すること(yaml,jsonの設定例はコード例ではないため、必要に応じて記述しても良い)
- 将来の拡張案は絶対に記述しないこと
- オプションなどの仕様や実装の選択肢は残さないで、具体的に明示すること
- 実装スケジュール/実装フェーズは絶対に記述しないこと
- markdown形式で記述し、複雑な説明にはmermaid形式の図を描くようにすること
- 作成,修正したら**絶対に、必ず**内容をレビューして内容に**矛盾はないか**、**冗長ではないか**などを確認、修正して**必ず**レビュー結果を報告してください

---

## レビュー（必須）

要件定義書を作成・修正した後は**必ず**以下のレビューを行い、結果をユーザーに報告すること。

1. 内容に矛盾がないか
2. 冗長な記述がないか
3. 上記の網羅性チェック、ドキュメント作成ルールを全て満たしているか

問題があれば修正してから報告する。問題がなければ完了とし、`docs/requirements.md` として保存する。
