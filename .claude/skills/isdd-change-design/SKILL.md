# isdd-change-design - 変更詳細設計スキル

## 事前確認（必須）

- **既存のコードベースがすでに存在するプロジェクトにこのスキルを適用する場合**: 先に `isdd-reverse-engineering` スキルを実行して要件定義書と設計書を生成・確定させること。

あなたはシステムエンジニアのスペシャリストとして振る舞い、変更要件定義書をもとに矛盾のない完全な変更設計書を作成する。

## 最重要制約

- 既存の docs/detail_design.md は絶対に変更しない
- ユーザーから変更要件定義書の指定がない場合は、対象の change_requirements.md を必ず確認する
- 入力の変更要件定義書は .history/[YYYYMMDD]-[タスク名]/change_requirements.md を前提とする
- 変更設計書は同ディレクトリの change_detail_design.md として保存する

## ID付与ルール（必須）

- 変更設計書に記載する全ての設計要素には、必ず `isdd-id-definitions` に従った設計ID（`DS-*`）を付与する
- 設計IDは対応する要件ID（`RQ-*`）と紐付けて採番する
- ID体系・フォーマット・命名規則は `isdd-id-definitions` を唯一の情報源として参照する
- IDなしで設計要素を記述することは絶対に禁止する

## 基本方針

- 作業前に docs/requirements.md と docs/detail_design.md を読んで既存仕様を理解する
- 変更点は既存設計との差分として整理し、変更がない項目は既存設計参照として明記する
- 削除する要素は必ず削除理由を記載する
- 不足情報があれば要件レベル不足のみユーザーに確認する
- 設計、検証、修正を繰り返し、完全性が担保されるまで継続する
- MVPに必要最小限の変更設計に限定する
- 同じ意味の処理の再実装を禁止し、共通化方針を明記する

## 出力形式

- markdown形式
- 図はmermaid
- コードは記述しない
- 日本語で具体的かつ漏れなく記載する
- 将来拡張案は記述しない
- 実装スケジュール、実装フェーズは記述しない

## 変更設計書に必須の内容

削除要素以外は以下の項目を記載し、変更なし項目は既存設計書参照と明記する。
各セクションに登場する全ての設計要素（クラス・API・関数・データモデル等）には、必ず `isdd-id-definitions` に従った `DS-*` IDを付与すること。IDのない設計要素は記述しない。

### 1. 言語・フレームワーク

- 言語、フレームワークの選定
- 特段の理由がなければPython
- 新たにGUIが必要な場合
  - 基本はStreamlit
  - 複雑画面はVue（Vuetify）+ FastAPI
- Vue + FastAPIの場合の必須記載
  - フロントはマルチステージビルド後Nginx配信
  - バックはNginxリバースプロキシ
  - エンドポイントは /api/ に統一

### 2. システム構成

- コンポーネント一覧
- システム構成図（mermaid）
- コンポーネント役割
- インターフェースとデータフロー
- ネットワーク構成図（mermaid）

### 3. データベース設計

- DB必要性の判定
- 必須でない場合は「DBは必須ではないため、DB設計は行わない」と明記
- DB選定ルール
  - 少量かつ単純アプリはSQLite
  - それ以外で必須の場合はPostgreSQL
- テーブル設計（カラム、型、制約）
- リレーション図（mermaid）

### 4. アーキテクチャ設計

- 外部設計
  - UI設計（GUIは画面一覧、遷移図、AAモック。CUIはコマンド仕様）
  - 外部システム連携（一覧、IF、データフロー）
  - 外部DB連携（一覧、IF、データフロー）
- 内部設計
  - 処理フロー図（mermaid）
  - 処理の役割
  - 必要に応じたバッチ設計

### 5. クラス設計

- 変更対象クラスの一覧と役割
- SOLID原則への適合状況
- 各クラスの責務、主要属性、主要メソッド
- クラス図（mermaid）
- メッセージ一覧、役割、フロー図（mermaid）

### 6. その他設計

- エラーハンドリング（想定エラー一覧）
- セキュリティ設計

### 7. コード設計

- ディレクトリ構成（AA）
- ファイル名、役割、含まれるクラスの一覧
- コーディング規約表

### 8. テスト設計

- テスト種別と内容
- 目的と方法
- 実装すべきテストケース
  - 全モジュール単体テスト
  - 全機能要件の結合テスト
  - 総合テスト
  - GUIやCUIがある場合のユーザー視点テスト

### 9. 運用設計

- docker compose起動前提
- 初期化が必要なら起動時自動化
- README.md への起動方法、操作説明記載方針

### 10. ログ・監視・アラート設計

- ログ要否判定
  - 不要なら「ログの設計は必須ではないため、ログの種類と内容の記述は行わない」と明記
- 監視、アラート要否判定
  - 不要なら「監視・アラートの設計は必須ではないため、監視・アラートの内容と対応方法の記述は行わない」と明記
- 必須の場合は内容と対応方法を表で定義

### 11. E2Eテスト設計

- 要件のテスト用利用シナリオを網羅
- 各シナリオに目的、前提、手順、期待結果を定義
- GUIの場合は全画面遷移を網羅しPlaywright前提で記述
- 外部連携がある場合、E2Eは `mock` モードと `real` モードの2モードを設計する
- 設計上は全E2Eシナリオを `mock` / `real` の両モードで実行対象として定義する
- docker composeに test_playwright サービスを追加する前提を記載
  - mcr.microsoft.com/playwright:v1.59.0 を使用
  - profile は test に加えて、外部連携時は mock / real の切替プロファイルを定義
  - 通常起動では起動しない
  - テストコードはマウントして即時反映
- E2E実行コマンドを設計に明記する
  - 実行例: `docker compose run --rm test_playwright sh -c "npm install && npx playwright test"`
- E2E運用方針を記載
  - e2e ディレクトリ配置
  - 外部モックは external/[システム名]/mock/ を正本とし、e2e 側はシンボリックリンクで参照する
  - シンボリックリンクはGit管理下に置く
  - モード切替は docker compose プロファイルとテスト実行コマンドの両方を提供する
  - テスト環境作成スクリプトを設計に含める
    - 原則はbashで実装する
    - 複雑処理はメインPJ言語、またはPythonを利用する
  - 実行URLはcompose内サービス名を使用
  - 実装時・CI時の必須ゴールは mock モードE2E全件通過とする
  - real モードE2Eは手動実行のみ必須とする
  - 失敗時は修正して再実行し全成功まで継続

## 完全性制約（必須）

既存設計書と合わせて以下を満たさない場合は不足として必ず指摘し修正する。

- 業務エンティティ一覧とCRUD、一覧、詳細、検索、状態管理
- エンティティと画面、API、クラスの対応関係
- トランザクション境界とロールバック条件
- 排他制御（楽観、悲観）
- API入出力、バリデーション、エラー仕様
- データ整合性（PK、FK、業務制約）
- 認証、認可、監査ログ
- ログ、監視、アラート、障害対応
- 各エンティティの状態遷移
- 全機能に対応する正常系、異常系テスト
- 同一意味処理の重複排除、共通化、共通モジュール配置
- 不要要素の列挙
- 削除しても要件を満たす要素の削除
- 全ての設計要素に `DS-*` IDが付与されているか（IDなし要素がある場合は不足として修正する）

---

## 実行ルール

- 設計、検証、修正を繰り返す
- 矛盾、不足は必ず指摘する
- 基本はユーザー確認なしで進める
- 外部連携先API仕様、外部スキーマなど要件不足は確認する
- 完全性確認後にレビューし、問題がなければ完了とする
- 完了時に .history/[YYYYMMDD]-[タスク名]/change_detail_design.md を保存する
- 保存後に .history/[YYYYMMDD]-[タスク名]/tasks.md を作成し、実装タスクを列挙する

## tasks.md 作成ルール

- 各タスクに完了条件を必ず定義する
- 各タスクにバリデーションタスクを必ず定義する
- 最終的に全変更点を実装済みか確認するタスクを必ず定義する
- 上記タスクの次に、変更要件定義書と変更設計書の内容を既存の docs/requirements.md と docs/detail_design.md に反映するタスクを必ず明記する

---

## ドキュメント作成ルール

- コード例は絶対に書かず、日本語で処理を具体的に記述すること（yaml、jsonの設定例はコード例ではないため、必要に応じて記述しても良い）
- 将来の拡張案は絶対に記述しないこと
- オプションなどの仕様や実装の選択肢は残さないで、具体的に明示すること
- 実装スケジュール／実装フェーズは絶対に記述しないこと
- markdown形式で記述し、複雑な説明にはmermaid形式の図を描くようにすること
- 作成、修正したら**絶対に、必ず**内容をレビューして内容に**矛盾はないか**、**冗長ではないか**などを確認、修正して**必ず**レビュー結果を報告してください

---

## レビュー（必須）

変更設計書を作成または修正したら、必ず以下をレビューし、結果を報告する。

1. 内容に矛盾がないか
2. 冗長な記述や冗長設計がないか
3. 完全性制約およびドキュメント作成ルールを全て満たしているか

問題があれば修正してから報告し、問題がなければ完了とする。
