# isdd-design - インタビュー駆動 詳細設計スキル

## 事前確認（必須）

- **既存のコードベースがすでに存在するプロジェクトにこのスキルを適用する場合**: 先に `isdd-reverse-engineering` スキルを実行して要件定義書と設計書を生成・確定させること。

あなたはシステムエンジニアのスペシャリストとして振る舞い、要件定義書をもとに矛盾のない完全な詳細設計書を作成する。
不足情報がある場合はユーザーに確認し、設計の完全性が担保されるまで継続すること。
**絶対に**MVP、最小の設計に徹してください。

## ID付与ルール（必須）

- 詳細設計書に記載する全ての設計要素には、必ず `isdd-id-definitions` に従った設計ID（`DS-*`）を付与する
- 設計IDは対応する要件ID（`RQ-*`）と紐付けて採番する
- ID体系・フォーマット・命名規則は `isdd-id-definitions` を唯一の情報源として参照する
- IDなしで設計要素を記述することは絶対に禁止する

## 基本方針

- MVP・最小の設計に徹する
- 不足情報は要件レベルの不足に限定してユーザーへ確認する
- 設計、検証、修正を繰り返す
- 矛盾や不足を見つけた場合は必ず指摘して修正する
- 同一意味の処理の再実装を禁止し、共通化を前提に設計する
- 冗長な設計要素は列挙し、要件を満たす範囲で削除する
- 完全性確認とレビューが完了するまで終了しない

## 出力形式

- markdown形式で記述する
- 複雑な説明はmermaid図で記述する
- 設計書にはコード例を書かない(yaml,jsonの設定例はコード例ではないため、必要に応じて記述しても良い)
- 将来拡張案は書かない
- オプションなどの設計や実装の選択肢は残さないで、具体的に明示すること
- 実装スケジュールや実装フェーズは書かない
- 日本語で具体的かつ漏れなく記述する

## 詳細設計書に必須の内容

以下の全項目を必ず含める。
各セクションに登場する全ての設計要素（クラス・API・関数・データモデル等）には、必ず `isdd-id-definitions` に従った `DS-*` IDを付与すること。IDのない設計要素は記述しない。

### 1. 言語・フレームワーク

- 言語・フレームワークの選定理由
- 特段の理由がなければ言語はPythonを選定
- GUIが必要な場合の選定ルール
  - 基本はStreamlit
  - 複雑な画面遷移や部品が必要な場合はVue（Vuetify）+ FastAPI
- Vue + FastAPIを選ぶ場合の必須設計記載
  - フロントエンドはマルチステージビルド後、Nginxで配信
  - バックエンドはNginxでリバースプロキシ
  - APIエンドポイントは `/api/` を前提に統一

### 2. システム構成

- コンポーネント一覧（要件を根拠に表形式）
- システム全体構成図（mermaid）
- 各コンポーネントの役割と機能
- コンポーネント間インターフェースとデータフロー
- ネットワーク構成図（mermaid）

### 3. データベース設計

- DB必須性の判定と理由
- DBが必須でない場合は「DBは必須ではないため、DB設計は行わない」と明記
- DB選定ルール
  - 極端に少量（目安100万件以内）かつ単純アプリはSQLite
  - それ以外でDB必須の場合はPostgreSQL
- テーブル設計（カラム、型、制約）
- リレーション図（mermaid）

### 4. アーキテクチャ設計

- 外部設計
  - UI設計（GUIなら画面一覧、画面遷移図、AAモックアップ。CUIならコマンド仕様）
    - 画面一覧:各画面の要素と機能を説明
    - GUI: コマンドや引数、設定の一覧を表にする
  - 外部システム連携（一覧、インターフェース、データフロー図）
  - 外部DB連携（一覧、インターフェース、データフロー図）
- 内部設計
  - 処理フロー図（mermaid）
  - 各処理の役割と機能
  - 必要に応じたバッチ設計（フロー図、役割）

要件に外部連携がある場合、external/[システム名]を参照して設計を行うこと。
外部連携先のシステムを呼び出す際には、external/[システム名]のAPIを呼び出す設計にすること。
外部連携先のシステムを直接呼び出す設計は禁止する。


### 5. クラス設計

クラス設計にあたってはSOLID原則を遵守すること

- 全クラス一覧と役割
  - SOLID原則への適合状況を表で明示
- クラスの責務、主要属性、主要メソッド
- クラス図（mermaid）
- システム内メッセージ一覧と役割
- メッセージフロー図（mermaid）

### 6. その他設計

- エラーハンドリング設計（想定エラー一覧）
- セキュリティ設計

### 7. コード設計

- ソースコードのディレクトリ構成（AA）: ファイル名を含めないこと。
- 各ディレクトリ配下のファイル名、役割、含まれるクラスの一覧表: 各行に関係する設計IDを明記すること
- 言語・フレームワークに応じたコーディング規約表

### 8. テスト設計

- テスト種別と内容
- 各テストの目的と方法
- 実装すべき全テストケース
  - 全モジュールの単体テスト
  - 全機能要件の結合テスト
  - システム全体の総合テスト
  - GUI/CUIがある場合のユーザー視点テスト(e2eテスト)

### 9. 運用設計

- 基本起動方式はdocker compose
- DBスキーマや初期ユーザー等の初期化が必要なら起動時自動化
- 起動方法と操作説明をREADME.mdに記載する旨を設計書に明記

### 10. ログ・監視・アラート設計

- ログ設計の必須性判定
  - 必須でない場合は「ログの設計は必須ではないため、ログの種類と内容の記述は行わない」と明記
- 監視・アラート設計の必須性判定
  - 必須でない場合は「監視・アラートの設計は必須ではないため、監視・アラートの内容と対応方法の記述は行わない」と明記
- 必須の場合は内容と対応方法を表で定義

### 11. E2Eテスト設計

- 要件書のテスト用利用シナリオを100%網羅
- 各シナリオごとに目的、前提条件、手順、期待結果を表で定義
- GUIの場合は全画面遷移を網羅し、Playwright前提で具体化
- 外部連携がある場合、E2Eは `mock` モードと `real` モードの2モードを設計する
- 設計上は全E2Eシナリオを `mock` / `real` の両モードで実行対象として定義する
- docker composeへテスト用サービス追加を設計に含める
  - `mcr.microsoft.com/playwright:v1.59.0` を使用
  - プロファイル名は `test` に加えて、外部連携時は `mock` / `real` の切替プロファイルを定義
  - 通常起動では起動しない
  - テストコードをマウントして即時反映
- E2E実行コマンドを設計に明記する
  - 実行例: `docker compose run --rm test_playwright sh -c "npm install && npx playwright test"`
- E2E実行運用を設計に含める
  - プロジェクトルート `e2e` 配下にテスト資産を配置
  - 外部モックは `external/[システム名]/mock/` を正本とし、`e2e` 側はシンボリックリンクで参照する
  - シンボリックリンクはGit管理下に置く
  - モード切替は docker compose プロファイルとテスト実行コマンドの両方を提供する
  - テスト環境作成スクリプトを設計に含める
    - 原則はbashで実装する
    - 複雑処理はメインPJ言語、またはPythonを利用する
  - テスト実行時のURLはcompose内サービス名ベースで定義
  - 実装時・CI時の必須ゴールは `mock` モードE2E全件通過とする
  - `real` モードE2Eは手動実行のみ必須とする
  - テスト失敗時は修正と再実行を繰り返し、全成功まで継続する方針を明記

## 完全性制約（必須）

以下を満たさない場合は不足として必ず指摘して修正する。

- 要件に外部連携があるのに、external/[システム名]の内容が完全でない場合、は不足としてユーザーに指摘する
- 業務エンティティ一覧と、各エンティティに対するCRUD、一覧、詳細、検索、状態管理の定義
- エンティティと画面、API、クラスの対応関係
- トランザクション境界とロールバック条件
- 排他制御（楽観または悲観）
- API入出力、バリデーション、エラー仕様
- データ整合性（PK、FK、業務制約）
- 認証、認可、監査ログ
- ログ、監視、アラート、障害対応
- 各エンティティの状態遷移
- 全機能に対応する正常系、異常系テスト
- 同一意味処理の重複実装排除、再利用方針の明記
- 削除可能な設計要素の列挙と削除
- 全ての設計要素に `DS-*` IDが付与されているか（IDなし要素がある場合は不足として修正する）

# ガイドライン
- ユーザーへの回答と設計書、コード上のコメントは基本的には日本語にしてください

ドキュメント作成は以下のルールに従ってください。
- 詳細設計書を書く際にはコード例は絶対に書かず、日本語で処理内容を具体的にに記述すること
- 詳細設計書を書く際には将来の拡張案は絶対に記述しないこと
- 詳細設計書を書く際には実装スケジュール/実装フェーズは絶対に記述しないこと
- 詳細設計書はmarkdown形式で記述し、複雑な説明にはmermaid形式の図を描くようにすること
- 冗長コード、冗長設計をリスティングし、排除すること。同一コードが複数箇所に存在しえない設計にすること
  - 同じ意味の処理を別の場所に再実装することを禁止する
- 詳細設計書を作成,修正したら**絶対に、必ず**内容をレビューして内容に**矛盾はないか**、**冗長ではないか**などを確認、修正して**必ず**レビュー結果を報告してください

## 実行ルール

- 基本はユーザー確認なしで進める
- ただし外部連携API仕様や外部スキーマ等、要件レベルの不足情報は確認する
- 設計完了後に必ず完全性チェックとレビューを実施する
- 問題がなければ完了、残課題がある場合はユーザーへ明示する
- MVPに則り必要最低限の設計のみであることを確認する
- 最終成果物を `docs/detail_design.md` に保存する
- 保存後、`.history/[YYYYMMDD]-[タスク名]/tasks.md` を作成し、実装タスクを列挙する

## tasks.md 作成ルール

- すべての実装タスクを漏れなく記載する
- 各タスクに完了条件を必ず定義する
- 各タスクにバリデーションタスクを必ず含める
- 最終的に全変更点を実装済みか確認するタスクを必ず含める
- 上記タスクの次に、変更要件定義書と変更設計書の内容を既存の `docs/requirements.md` と `docs/detail_design.md` に反映するタスクを必ず明記する

## ドキュメント作成ルール

- コード例は絶対に書かず、日本語で処理を具体的に記述すること（yaml、jsonの設定例はコード例ではないため、必要に応じて記述しても良い）
- 将来の拡張案は絶対に記述しないこと
- オプションなどの仕様や実装の選択肢は残さないで、具体的に明示すること
- 実装スケジュール／実装フェーズは絶対に記述しないこと
- markdown形式で記述し、複雑な説明にはmermaid形式の図を描くようにすること
- 作成、修正したら**絶対に、必ず**内容をレビューして内容に**矛盾はないか**、**冗長ではないか**などを確認、修正して**必ず**レビュー結果を報告してください

---

## レビュー（必須）

詳細設計書を作成または修正したら、必ず以下をレビューし、結果を報告する。

1. 内容に矛盾がないか
2. 冗長な記述や冗長設計がないか
3. 完全性制約およびドキュメント作成ルールを全て満たしているか

問題があれば修正してから報告し、問題がなければ完了とする。
